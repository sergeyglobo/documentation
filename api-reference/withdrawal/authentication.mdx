## Аутентификация запросов к API

Для аутентификации запросов используется [Basic HTTP authentication scheme (RFC 7617)](https://datatracker.ietf.org/doc/html/rfc7617). Таким образом обязательной частью учетных данных является пара `username` \+ `password` (выдается менеджером интеграции), которые должны быть переданы в каждом запросе к API в заголовке `Authorization`:

```text
Authorization: Basic base64(key:password)
```

Каждая интеграция одновременно может использоваться несколько наборов учетных данных, это может применяться для их ротации.

Для проверки доступности API и учетных данных можно использовать метод API `/public/api/withdraw/generic/v2/check_health`:

```json
{
  // тело запроса - произвольный JSON объект
}
```

Успешный ответ:

```json
{
  "success": true,
  "trace_id": "d33359866ec888f426d2bee748b76ff6"
}
```

Ошибка аутентификации:

```json
{
  "success": false,
  "trace_id": "3acb855a8c75712b4eff29900fed4f1a",
  "error": {
    "code": "UNAUTHORIZED",
    "details": "authentication failed"
  }
}
```

В дополнении к проверке корректности пары `username` \+ `password` аутентификация запросов может быть дополнительно усилена проверкой IP адреса вызывающей стороны и проверкой подписи запроса.

### Ограничение допустимых IP адресов отправителя

При необходимости _можно_ согласовать список IP адресов, с которых будут выполняться вызовы API. Запросы с других адресов будут отклоняться даже при правильных учетных данных.

### Подпись запросов

В качестве дополнительной меры безопасности можно использовать подпись запросов с использованием асимметричной криптографии. Для этого необходимо сгенерировать пару ключей `key_prv` \+ `key_pub` и предварительно передать менеджеру интеграции публичный ключ `key_pub`, получив в ответ идентификатор этого ключа `key_id`.

Используемые ключи могут быть `RSA` или `ECDSA` с кривой `prime256v1`/`secp256r1`/`nist-p256`. Рекомендуется использовать `ECDSA`.

Для каждой интеграции может быть привязано несколько действующих одновременно ключей, это может применяться для их ротации.

Алгоритм подписи различается для `RSA` и `ECDSA`.

Процедура подписи запроса с RSA ключом следующая:

- выбирается используемый RSA ключ и добавляется заголовок `X-RSA-KEY-ID: <key_id>`
- генерируется уникальная случайная строка (для каждого запроса) `nonce` и добавляется заголовок `X-RSA-NONCE: <nonce>`

- добавляется заголовок `X-RSA-SHA256-SIGN: <sign>`

## Аутентификация уведомлений

Для аутентификации уведомлений используется подпись тел HTTP запросов с использованием асимметричной криптографии. Подпись каждого сообщения выполняется с помощью двух алгоритмов (`ECDSA-SHA256` и `RSA-SHA256`), принимающая сторона может проверять подпись по одному из алгоритмов исходя из своих технических возможностей. Каждый подписанный запрос содержит идентификаторы используемых для подписи ключей в заголовках HTTP сообщения.

Получить публичные ключи подписи запросов можно у менеджера интеграции.

Помимо проверки подписи сообщений принимающая сторона может проверять IP адрес отправителя (белый список), запросы сервиса приходят с заранее обговоренного набора IP адресов.

### RSA-SHA256

Информация о RSA подписи находится в заголовках HTTP сообщения:

- `X-RSA-KEY-ID` - идентификатор используемого `RSA` ключа
- `X-RSA-NONCE` - случайная строка `nonce`, уникальная для каждого запроса
- `X-RSA-SHA256-SIGN` - подпись сообщения в кодировке `base64`

Алгоритм вычисления подписи:

1. выбирается используемый `RSA` ключ и записывается заголовок `X-RSA-KEY-ID`
2. генерируется случайная строка и записывается в заголовок `X-RSA-NONCE`
3. формируется `body` - бинарное представление **сериализованного тела** HTTP сообщения в кодировке `UTF-8` - то что будет отправлено по сети, после сериализации
4. вычисляется подпись `sign` для сообщения `nonce`\+`body` с помощью `RSA` с алгоритмом хеширования `SHA256` и форматом подписи `PKCS#1 v1.5`
5. полученная подпись `sign` кодируется в `base64` и записывается в заголовок `X-RSA-SHA256-SIGN`

Алгоритм проверки подписи:

1. по значению заголовка `X-RSA-KEY-ID` выбирается публичный `RSA` ключ (может быть автоматически получен вызовом CardRouting API, ключи следует кешировать)
2. берется значение `nonce` из заголовка `X-RSA-NONCE`
3. берется `body` - бинарное представление **сериализованного тела** HTTP сообщения в кодировке `UTF-8` - то что получено по сети, до десериализации
4. значение `X-RSA-SHA256-SIGN` преобразуется из `base64` в бинарное представление `sign`
5. выполняется проверка подписи `sign` для сообщения `nonce`\+`body` с помощью `RSA` с алгоритмом хеширования `SHA256` и форматом подписи `PKCS#1 v1.5`
6. сообщение считается аутентифицированным если проверка подписи прошла успешно

### ECDSA-SHA256

Для ECDSA используется кривая `prime256v1` (также известна как `secp256r1` и `NIST P256` ).

Информация о ECDSA подписи находится в заголовках HTTP сообщения:

- `X-ECDSA-KEY-ID` - идентификатор используемого `ECDSA` ключа

Алгоритм вычисления подписи:

1. выбирается используемый `ECDSA` ключ и записывается заголовок `X-ECDSA-KEY-ID`
2. формируется `body` - бинарное представление **сериализованного тела** HTTP сообщения в кодировке `UTF-8` - то что будет отправлено по сети, после сериализации
