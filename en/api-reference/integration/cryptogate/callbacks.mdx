### Notification Authentication


<Steps>
  <Step title="Verify signature">
  For security, it is necessary to verify the notification signature.

<Expandable title="RSA-SHA256" defaultOpen="true">
  RSA signature information is located in the HTTP message headers:

  - `X-RSA-KEY-ID` - ID of the used `RSA` key
  - `X-RSA-NONCE` - random `nonce` string, unique for each request
  - `X-RSA-SHA256-SIGN` - message signature in `base64` encoding

  #### **Signature Calculation Algorithm:**

  1. The used `RSA` key is selected, and the `X-RSA-KEY-ID` header is written
  2. A random string is generated and written to the `X-RSA-NONCE` header
  3. `body` is formed - binary representation of the **serialized body** of the HTTP message in `UTF-8` encoding - what will be sent over the network after serialization
  4. The signature `sign` is calculated for the message `nonce`+`body` using `RSA` with `SHA256` hashing algorithm and signature format `PKCS#1 v1.5`
  5. The resulting signature `sign` is encoded in `base64` and written to the `X-RSA-SHA256-SIGN` header

  #### **Signature Verification Algorithm:**

  1. Select the public `RSA` key by the value of the `X-RSA-KEY-ID` header (can be automatically obtained by calling CardRouting API, keys should be cached)
  2. Take the `nonce` value from the `X-RSA-NONCE` header
  3. Take `body` - binary representation of the **serialized body** of the HTTP message in `UTF-8` encoding - what was received over the network, before deserialization
  4. The value of `X-RSA-SHA256-SIGN` is converted from `base64` to binary representation `sign`
  5. The signature `sign` is verified for the message `nonce`+`body` using `RSA` with `SHA256` hashing algorithm and signature format `PKCS#1 v1.5`
  6. The message is considered authenticated if the signature verification pass
</Expandable>

<Expandable title="ECDSA-SHA256">
  For ECDSA, the `prime256v1` curve (also known as `secp256r1` and `NIST P256`) is used.

  ECDSA signature information is located in the HTTP message headers:

  - `X-ECDSA-KEY-ID` - ID of the used `ECDSA` key
  - `X-ECDSA-SHA256-SIGN` - message signature in `base64` encoding

  #### **Signature Calculation Algorithm:**

  1. The used `ECDSA` key is selected, and the `X-ECDSA-KEY-ID` header is written
  2. `body` is formed - binary representation of the **serialized body** of the HTTP message in `UTF-8` encoding - what will be sent over the network after serialization
  3. The signature `sign` is calculated for the message `body` using `ECDSA` with `SHA256` hashing algorithm and signature format `IETF RFC 3279`
  4. The resulting signature `sign` is encoded in `base64` and written to the `X-ECDSA-SHA256-SIGN` header

  #### **Signature Verification Algorithm:**

  1. Select the public `ECDSA` key by the value of the `X-ECDSA-KEY-ID` header (can be automatically obtained by calling CardRouting API, keys should be cached)
  2. Take `body` - binary representation of the **serialized body** of the HTTP message in `UTF-8` encoding - what was received over the network, before deserialization
  3. The value of `X-ECDSA-SHA256-SIGN` is converted from `base64` to binary representation `sign`
  4. The signature `sign` is verified for the message `body` using `ECDSA` with `SHA256` hashing algorithm and signature format `IETF RFC 3279`
      5. The message is considered authenticated if the signature verification pass
</Expandable>

### Notification Body Parameters

    | **Parameter**              | **Type**       | **Description** |
    | :------------------------ | :------------ | :----------- |
    | `reference_id`           | string        | Globally unique operation identifier assigned by the service. You should save it and use it to confirm the operation. |
    | `user_id`                | string        | User identifier on the project side. |
    | `result`                 | string        | Final result of operation processing (`complete`, `fail` etc.). For deposits to be credited, the value is `complete`. |
    | `current_status`         | string        | Current status of the operation. For successful deposits — `success`. |
    | `payment_product`        | string        | Payment product code, e.g. `cryptogate` or `cryptogate_direct`. |
    | `payment_method_type`    | string        | Payment method type. For crypto deposits — `crypto_deposit`. |
    | `payment_custom_data`    | any \| null   | Additional data passed when creating the operation. Can be `null`. |
    | `deposit.amount`         | number        | Amount to be credited to the user's balance. |
    | `deposit.currency`       | string        | Deposit currency (e.g. `TRX`, `USDC`). |
    | `crypto[]`               | array         | Array of objects with detailed information about the crypto transaction. |
    | `crypto[].amount`        | number        | Actually received amount in the network. |
    | `crypto[].currency`      | string        | Transaction cryptocurrency (e.g. `TRX`, `USDC`). |
    | `crypto[].oper_type`     | string        | Operation type, for deposits — `deposit`. |
    | `crypto[].status`        | string        | Crypto transaction status, usually `success` upon successful crediting. |
    | `crypto[].to_address`    | string        | Crypto address to which the user sent funds. |
    | `crypto[].crypto_txn_id` | string        | Transaction identifier in the blockchain. |
    | `rates`                  | object        | Currency pair rates at the time of the operation. Key — currency pair (`TRX-EUR`, `TRX-USD`, etc.), value — numeric rate. |

   #### Notification Resending
<Info>
  If your service returns response code 200 or 204 in response to the callback, the notification is considered successfully received.
</Info>
If a response code other than 200 or 204 is received, 10 retry attempts will be made with increasing intervals.
